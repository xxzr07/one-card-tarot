# 1. このアプリにおける「デッキ」の定義

このアプリのデッキとは、Rider–Waite–Smith（以下RWS）の78カード識別体系を共有しながら、画像・解釈・文章体験を独立して持つコンテンツ単位である。

## 1.1 4層の分離

| 層 | 内容 | 現在の状態 |
|---|---|---|
| A. アプリ全体 | TODAY / HISTORY / SETTINGS、1日1回、抽選、正逆、IndexedDB、PWA、デッキ選択 | デッキコンテンツから分離済み |
| B. カード識別・RWS基盤 | `cardId`、番号、英語名、スート、ランク、RWS主要象徴 | `data/rws-cards.json` に78件を実装 |
| C. Deck 01 | 78画像、裏面、視覚モチーフ、正逆別のキーワード・解説・問い | `decks/deck-01/` に独立パッケージとして実装 |
| D. 新規デッキ制作物 | 独自画像、裏面、視覚モチーフ、キーワード、解説、問い、デッキ説明 | `decks/_template/`から開始可能 |

## 1.2 共通基盤と作品部分

新しいデッキはDeck 01の色違いではない。再利用するのは、カードID、抽選・保存・表示のロジック、画像サイズなどの技術契約である。新規制作するのは、ビジュアル、象徴の翻案、キーワード、解説、正逆別の問い、カード裏面、デッキの名称・説明・世界観である。

## 1.3 RWS準拠部分とデッキ固有部分

### RWS準拠として共通に維持する部分

- 78枚の構成: 大アルカナ22枚、WANDS / CUPS / SWORDS / PENTACLES各14枚。
- カードを一意に識別する `cardId`。
- 大アルカナの番号と順序。
- 小アルカナのスート、数札、コートカードの区分。
- RWS上の主要象徴と、カード識別に必要な意味上の核。
- 正位置／逆位置という2方向。

### デッキごとに独自制作する部分

- 画風、配色、線、質感、人物、顔、背景、空間、描画密度。
- RWS象徴の残し方・抽象化・置換方法。
- 表面78枚と共通裏面1枚。
- デッキ表示名、サブタイトル、説明、代表カード。
- 視覚モチーフの説明。
- 正位置／逆位置のキーワードと解説。
- 正位置／逆位置の「今日の問い」。
- 必要に応じた日本語表示名やデッキ固有の補助表示。

［現行実装］キーワード、解説、問いはすべて`deck.json`のカード×正逆へ移行済みである。

# 2. アプリ共通仕様

## 2.1 TODAY

［現行実装］

- 端末の現地日付を `YYYY-MM-DD` にし、1日1レコードをIndexedDBへ保存する。
- 同日のレコードがあれば引き直さず、保存済みのカードを表示する。
- ドロー前だけデッキを選択できる。選択インデックスはメモリ上のみで、ドロー前の選択状態自体は永続化されない。
- 抽選対象は、RWS共通データのうち選択デッキに同じ `cardId` が存在するカード。
- カードは1枚をランダム抽選し、方向も正位置／逆位置から独立にランダム抽選する。
- `crypto.getRandomValues()` が利用できる場合はそれを使用し、利用できない場合のみ `Math.random()` を使用する。
- ドロー後は選択デッキ固有のカード画像、方向別キーワード・解説・問いを表示する。
- 複数デッキが同じ `cardId` を持つ場合、同じ履歴カードを別デッキの画像と問いで見る `ALTERNATE VIEW` の土台がある。

［アプリ共通］

- 1日1回、端末現地時間0:00で日付を切り替える。
- 抽選結果の `cardId` と `orientation` は、表示デッキやUI変更から独立した不変値として扱う。
- 裏面表示中に抽選結果を視覚的に漏らさない。
- 表面だけを逆位置時に180度回転する。

## 2.2 デッキ選択

［現行実装］

- `decks/index.json`から`enabled`なデッキmanifestを読み込み、読込後の配列を`window.DECKS`として参照する。
- 矢印で `selectedDeckIndex` を循環させる。
- デッキ順は`decks/index.json`の登録順。
- `window.DECKS.length === 1` のとき矢印は無効になる。
- 選択中のデッキ名とサブタイトルを表示する。

- デッキ順、公開／非公開、初期選択は `decks/index.json` で管理する。
- デッキIDは保存データとの互換性に関わるため、公開後に変更・再利用しない。
- 一度公開したデッキを削除する場合も、履歴復元用の最小メタデータとアセットは残す。

## 2.3 ランダムドローと正逆

［現行実装］

既存version 1:

```js
{
  version: 1,
  date: "YYYY-MM-DD",
  cardId: "major-09",
  orientation: "upright" | "reversed",
  deckId: "deck-01",
  createdAt: "ISO-8601 timestamp"
}
```

新規version 2は上記に`deckContentVersion`と、表示時点の文章` snapshot`を加える。IndexedDBのDB versionは`1`、object storeとkeyPathも変更していない。

- カード抽選は対象配列から1件。
- 正逆は50%ずつ。
- `date` がIndexedDBのキーなので、1日につき保存できる正式レコードは1件。

## 2.4 HISTORY

［現行実装］

- 月間カレンダー方式。
- レコードがある日だけ選択可能。
- 選択するとTODAY表示領域で過去カードを再表示する。
- 過去表示は記録された `deckId` を使い、version 2ではsnapshot文章を優先する。
- version 1履歴は現在の同デッキデータから復元する。
- 「今日のカードへ戻る」操作がある。

## 2.5 SETTINGS / バックアップ

［現行実装］

- IndexedDBの全レコードをJSONでエクスポートする。
- schemaVersion 1 / 2のJSONを検証してインポートする。
- インポート条件は、日付形式、既知の `cardId`、既知の `deckId`、正しい `orientation`。
- 外部送信処理はなく、保存先はブラウザ内。

## 2.6 PWA

［現行実装］

- `manifest.webmanifest`、`service-worker.js`、アプリアイコンを使用。
- standalone、portrait、相対パス構成でGitHub Pagesに配置可能。
- iPhone safe areaとreduced motionに対応。
- アプリシェルはキャッシュ優先。
- `decks/index.json`と`decks/`配下はネットワーク優先、失敗時キャッシュ。
- Service Workerは一覧から各manifest、裏面、78画像を動的にprecacheする。
- 現行テーマはMATTE版の`#0b0c0f`。

# 5. 今日の問いの構造

## 5.1 現在の保存場所と名称

［現行実装］問いは共通カードではなく、`deck.json`の次の2箇所にある。

- `cards[cardId].upright.question`
- `cards[cardId].reversed.question`

同じ `cardId` でもデッキごとに異なる文章を持てる。この要件は現在すでに満たしている。

表示時は、履歴の `orientation` に応じて次のように選ぶ。

```js
deck.cards[reading.cardId][reading.orientation].question
```

## 5.2 方向別構造

キーワード・解説・問いは次の入れ子構造へ統一済み。

```js
upright: { keywords, meaning, question }
reversed: { keywords, meaning, question }
```

旧`uprightQuestion / reversedQuestion`形式はアプリ本体から廃止済み。

## 5.3 履歴との関係

version 2履歴はドロー時点の問い・キーワード・解説・名称をsnapshot保存する。
既存version 1履歴だけは`deckId + cardId + orientation`から現在のデッキデータを参照して復元する。

# 6. 今日の問いの文章制作ルール

## 6.1 Deck 01から確認できた定量的特徴

- 156問すべて一文・一問形式。
- 文字数は18〜31文字、平均24.3文字、中央値24文字。
- 156問すべてが `だろう。` または `だろうか。` で閉じる。
- 正位置と逆位置で同一の問いは0件。
- 全文が重複する問いは0件。
- 命令、助言、予言、回答入力の要求はない。
- 画像内の物・形・距離・方向・光・影・余白を一つの入口にする。

## 6.2 Deck 01で確立した文章設計

［Deck 01固有］

1. RWSの基本意味を確認する。
2. Deck 01で強調した視覚モチーフを確認する。
3. 完成画像の具体物を一つ拾う。
4. 具体物から、答えを限定しない短い観察の問いへ変換する。
5. 正位置は象徴が開く・動く・結ぶ・育つ方向を見る。
6. 逆位置は象徴が滞る・閉じる・偏る・過不足になる方向を見る。
7. 正逆を単純な肯定／否定や幸運／不運にしない。

例:

- THE HERMIT 正位置: 「静けさの中でしか見えない灯りは、どこにあるだろう。」
- THE HERMIT 逆位置: 「暗がりに留まることと、目を閉ざすことの境目はどこだろう。」

## 6.3 すべてのデッキに推奨する品質基準

- 一つの問いに論点を一つだけ置く。
- 占い結果として未来・性格・他者の意図を断定しない。
- 「〜すべき」「〜しよう」「〜してください」など行動を強制しない。
- 過度にポジティブ／ネガティブへ誘導しない。
- 逆位置を罰・失敗・欠陥として扱わない。
- 答えを内蔵した誘導質問にしない。
- 抽象語だけで終わらせず、日常の経験へ接続できる焦点を置く。
- カードの意味、画像の象徴、問いの3点が説明可能な形でつながる。
- 同じ構文の機械的反復を避ける。
- 正逆の差を、焦点・状態・関係の違いとして明確にする。

## 6.4 新規デッキごとに定義する文章方針

制作開始前に必ず記入する。

- **問いのコンセプト**: 例）内省、行動選択、他者との距離、創作、身体感覚など。
- **問いの語調**: 静か、端的、対話的、観察的など。
- **問いの抽象度**: 具体寄り／中間／抽象寄り。
- **問いが向ける視点**: 自分、状況、関係、時間、選択、環境など。
- **正位置と逆位置の差**: 開閉、過不足、速度、距離、視野など、デッキ全体の軸。
- **避ける表現**: 断定、命令、診断、罪悪感、スピリチュアルな確言など。
- **長さの目安**: Deck 01を踏襲する場合18〜31文字。別設計なら上限を先に決め、アプリ表示を実機確認する。

# 7. 画像制作仕様

## 7.1 アプリ共通の技術契約

［現行実装］

| 項目 | 仕様 |
|---|---|
| 枚数 | 78枚 |
| 画像寸法 | 612×1206px |
| アスペクト比 | 306:603（約1:1.9706） |
| 形式 | PNG、8-bit RGBまたはRGBA |
| 表示 | CSS背景画像、`background-size: cover`、中央配置 |
| 表示枠 | 元画像と同じ比率なので通常はトリミングなし。CSS側で角丸クリップ |
| 逆位置 | 表面の `.tarot-card` をCSSで180度回転 |
| アプリ上の文字 | 番号・英語名はカード画像外にも表示。画像内文字はアセットの一部 |

新しいデッキでも612×1206pxのPNGを維持するのが最小変更である。WebPへ変更する場合は、manifest、検証、キャッシュ、画像生成ツールを同時に変更する。

## 7.2 ファイル命名規則

ファイル名は `cardId + .png` とする。

### 大アルカナ

- `major-00.png` 〜 `major-21.png`
- 2桁ゼロ埋め。
- Strengthは `major-08`、Justiceは `major-11` という現行RWS順を維持。

### 小アルカナ

- スート: `wands` / `cups` / `swords` / `pentacles`。
- ACE: `{suit}-ace.png`
- 2〜10: `{suit}-02.png` 〜 `{suit}-10.png`
- Court: `{suit}-page.png` / `-knight.png` / `-queen.png` / `-king.png`

例: `wands-ace.png`、`cups-03.png`、`swords-10.png`、`pentacles-queen.png`。

## 7.3 Deck 01固有の画面内レイアウト

以下はDeck 01の制作テンプレートであり、他デッキへの必須条件ではない。

- カードベース角丸: 画像生成テンプレート上 `rx=30`。
- 内側金枠: `x=31, y=31, width=550, height=1144, rx=24, stroke=1.6`。
- 図像クリップ: `x=32, y=32, width=548, height=1142, rx=23`。
- 番号: 上部中央、基準 `x=306, y=91`。
- 英語名: 下部中央、基準 `x=306, y=1080`。
- 補助✦: 下部中央、基準 `x=306, y=1122`。
- 明るいベース: `#f2eee6`を中心とする紙色。
- 暗背景の小アルカナ: `swords-09`、`swords-10`、`pentacles-05`。
- 文字と枠は生成画像に任せず、共通SVGテンプレートで合成。
- 大アルカナ画像は多くがRGB、小アルカナはRGBA。アプリ側のCSS角丸で最終表示をクリップする。

## 7.4 カード内文字の扱い

［現行実装］Deck 01はカード内に番号・英語名・✦を焼き込んでいる。アプリはカード内部へ文字を重ねない。

新規デッキでは次を企画時に決める。

- カード内文字を持つか。
- 英語名、日本語名、番号、スート記号のどれを載せるか。
- 文字を画像生成へ任せず、全78枚で同じテンプレートから合成するか。
- 長い名前でも字間・サイズ・余白が破綻しないか。

カード内文字を使わないデッキも技術上は可能だが、アプリ外で画像単体を見たときの識別性を別途検討する。

## 7.5 画像制作時の共通QA

- 612×1206pxである。
- 画像が破損せず読み込める。
- `cardId` とファイル名と内容が一致する。
- RWS上の数量、スート、人物数、象徴が誤っていない。
- 円、五芒星、柱、剣、塔など精度が必要な形を確認する。
- 角丸の外側、透過、背景、枠、カード名領域を100%表示と縮小一覧の両方で確認する。
- 大アルカナと小アルカナを78枚一覧で見て、描画密度・余白・色・質感の外れ値を確認する。
- 生成画像を直接採用せず、文字・幾何形・数量・中心軸を編集可能な工程で補正する。

# 8. カード裏面仕様

## 8.1 現在の実装

各デッキの`deck.json`が`backImage`を1件持つ。Deck 01は
`decks/deck-01/back.png`（612×1206px PNG）を使用する。
デッキ選択時にアプリが裏面を切り替え、正逆クラスは表面にだけ付与する。

## 8.2 Deck 01

Deck 01は「天体の軌跡」を採用。月・星・縦軸・波形・金枠・低彩度ラベンダー、
同一デッキ78枚共通、ドロー前は抽選結果に関係なく常に同じ上向きで表示する。

## 8.3 全デッキ共通の技術仕様

- 各デッキに `backImage` を1枚持たせる。
- 同じデッキの78枚では共通の裏面を使う。
- 選択デッキを切り替えた時点で裏面も切り替える。

- 画像サイズ・形式・角丸は表面と同じ技術契約にそろえる。
- UI側で枠、グロー、色被せなどを追加せず、裏面画像そのものを表示する。

## 8.4 新規裏面の検討項目

- 表面デザインとの整合性。
- 背景色。
- 中心モチーフと補助モチーフ。
- 線、粒子、紙、布、金属などの質感。
- 180度回転時の見え方。
- 中心ずれ、円の歪み、枠の左右差。
- 表面のカードや正逆を推測できる情報がないこと。
- iPhone上の縮小表示でも潰れないこと。
- 同じ上向きで表示する実装テスト。

# 10. HISTORY / 保存データとの関係

## 10.1 現行保存構造

IndexedDB:

- DB名: `one-card-tarot`
- version: `1`
- object store: `readings`
- keyPath: `date`

1件の保存内容:

| フィールド | 保存 | 用途 |
|---|---:|---|
| `version` | ✓ | レコード形式 |
| `date` | ✓ | 日付・主キー |
| `cardId` | ✓ | カード特定 |
| `orientation` | ✓ | 正逆特定 |
| `deckId` | ✓ | 元デッキ特定 |
| `createdAt` | ✓ | 作成時刻 |
| 問い本文 | v2のみ | v2はsnapshot、v1は現在のデッキデータから再取得 |
| キーワード／解説 | v2のみ | v2はsnapshot、v1は現在のデッキデータから再取得 |
| 画像 | × | 現在のデッキデータから再取得 |
| デッキ版 | v2のみ | `deckContentVersion` |

## 10.2 version 1互換

既存version 1レコードは削除・一括変換しない。`deckId / cardId / orientation / date`から
現在の登録デッキを参照して表示し、エクスポート時にもversion 1のまま保持する。

## 10.3 現行version 2

履歴は「ドロー時点の文章を残す」方式を推奨する。

```js
{
  version: 2,
  date: "YYYY-MM-DD",
  createdAt: "...",
  deckId: "deck-02",
  deckContentVersion: "1.0.0",
  cardId: "major-09",
  orientation: "reversed",
  snapshot: {
    deckName: "DECK 02",
    cardNumber: "IX",
    cardName: "THE HERMIT",
    keywords: ["..."],
    meaning: "...",
    question: "..."
  }
}
```

表示ルール:

1. 元デッキでの履歴本文は `snapshot` を優先する。
2. 画像は現在の`deckId + cardId`から読む。画像自体はsnapshotしない。
3. 旧version 1レコードは現行データから解決し、破壊的に書き換えない。
4. デッキIDは変更・再利用しない。コンテンツ変更時は`contentVersion`を上げる。
5. インポートは既知の`deckId`と`cardId`を要求し、version 1 / 2の両方を受け付ける。

画像を履歴へBase64保存すると容量が大きくなるため避ける。画像改訂まで完全固定したい場合は、版ごとの不変アセットパスを採用する。

# 14. アプリ組み込み手順

## 14.1 現行基盤への追加

1. `decks/_template/` を `decks/deck-XX/` へコピーする。
2. `cards/` に命名規則どおり78枚を置く。
3. `back.png` を置く。
4. `deck.json` にメタデータと78件の文章データを入れる。
5. `decks/index.json` に `{ "id": "deck-XX", "enabled": true, "manifest": "./deck-XX/deck.json" }` を1件追加する。
6. `node tools/validate-deck.cjs deck-XX` を実行する。
7. 78枚一覧を生成して目視QAする。
8. `node tests/validate.cjs`とiPhone相当テストを実行する。
9. オンラインで一度起動し、Service Workerが新デッキ78枚と裏面をキャッシュすることを確認する。

## 14.2 変更してはいけない箇所

通常の新規デッキ追加では`app.js`、`index.html`、`styles.css`、`service-worker.js`を編集しない。
旧`window.DECKS.push()`形式や`assets/deck-XX/`形式へ戻さない。

## 14.3 組み込み時の自動検証要件

validatorは最低限、次をエラーとして扱う。

- デッキIDの重複。
- 78枚未満／超過。
- 不明または重複した `cardId`。
- 画像欠落、寸法・形式不一致。
- 裏面欠落。
- 正逆キーワード・解説・問いの空欄。
- 正逆の問いが完全一致。
- 画像パスがデッキフォルダ外を誤参照。
- `cardId` と画像ファイル名の不一致。
- index未登録または無効化状態。
- HISTORYで参照する公開済みデッキIDの削除。

- **Deck ID**: `deck-01`
- **デッキ表示名**: `DECK 01`
- **サブタイトル**: `QUIET GEOMETRY`
- **短い説明**: 余白、光と影、幾何学によってRWSの象徴を静かに再構成するデッキ。
- **一文コンセプト**: RWSの主要象徴を読み取れる範囲で残し、人物中心の場面を余白・幾何学・光と影・自然物・痕跡へ置き換える。
- **想定する体験**: 占断ではなく、象徴と短い問いに静かに触れて思考を観察する。
- **代表カードID**: `major-18`
- **RWSから変更しない部分**: 78枚構成、名称、番号、スート、主要象徴、カードの意味上の核。
- **独自解釈する部分**: 人物動作を向き・距離・高低差・光・痕跡へ変換する。
- **避ける表現**: 人物の顔を中心にした物語説明、象徴の詰め込み、過剰な金、均一なフラット塗り。

- **基本色**: Paper `#F2EEE6`、Charcoal `#242426`。
- **アクセント色**: Grey mauve `#A9A0AC`、Deep mauve `#706875`、Muted gold `#AA8846`。
- **質感**: 紙と鉱物、ごく弱い粒子、半透明の水彩層、浅い影。
- **線**: 極細。輪郭をすべて閉じず余白へ消す。
- **陰影**: 低コントラストを基本とし、暗背景カードのみ深い面を使う。
- **人物表現**: 原則描かない。必要な場合は顔のないシルエット、背面、持ち物、足跡。
- **顔の表現**: 原則なし。
- **背景**: 静かな余白を25〜45%程度残す。
- **モチーフ**: 主象徴1つ＋補助象徴2〜4つ。
- **大アルカナ**: 各RWSアンカーを幾何形と自然物へ抽象化。
- **小アルカナ**: 数だけ並べず、RWSの関係を距離・高低差・整列・欠落・移動へ変換。
- **スート規則**: WANDS＝木・芽・火、CUPS＝器・流れ・水、SWORDS＝直線・風、PENTACLES＝正円・正五芒星・土。
- **Court規則**: PAGE＝小さな淡色形、KNIGHT＝馬と斜線、QUEEN＝包む形と有機物、KING＝高い石の座と中心軸。
- **カード内文字**: 上部番号、下部英語名、その下に小さな✦。共通テンプレートで合成。
- **避ける表現**: 歪んだ円、意図しない非対称、独立したタイトルパネル、スートごとに別デッキへ見える色分け。
